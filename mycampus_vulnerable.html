How it works
The attacker sends fake ARP reply packets to Sarah's laptop: "The router's IP (192.168.1.1) is at MY MAC address." Sarah's ARP table is updated. All her traffic is now delivered to the attacker first. The attacker enables IP forwarding so packets are relayed onward — neither Sarah nor the router notices anything unusual.

⚡ What Went Wrong
OWASP A02 · WSTG-CRYP-03
No HTTPS
All traffic — credentials and session cookie — travelled in plaintext. This is the root cause. Everything else flows from here.
OWASP A07 · WSTG-SESS-02
Cookie had no Secure flag
The Secure flag would have blocked the cookie over HTTP. Without it, it was freely readable by the attacker.
OWASP A04 · WSTG-ATHN-06
No re-auth for sensitive changes
Recovery email changed with only a session cookie. No current password was required.
User — No VPN
Public Wi-Fi without protection
A VPN would have encrypted Sarah's traffic even on the poisoned network, rendering the MitM useless.
