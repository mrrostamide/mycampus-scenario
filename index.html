The Scenario

Sarah is a university student. On a Tuesday afternoon she visits CaféBean, connects to the free Wi-Fi, and opens her laptop. She navigates to http://mycampus.com to check her semester grades. She types her username and password, spends a few minutes on her profile, then closes the laptop and leaves.

A week later she tries to log in again — her password doesn't work. She clicks "Forgot password" and waits for a reset email that never arrives. Digging through old notifications, she finds one she almost missed: "Your account email address has been changed." She never made that change. Her account now shows a recovery email she does not recognise.

She installed nothing. She visited no suspicious links. She only checked her grades. How did this happen?

Q1 — AttackARP poisoning → Man-in-the-Middle → Session hijacking
What happened: The attacker was physically on the same café LAN. They broadcast forged ARP reply packets to both Sarah's laptop and the router, poisoning both ARP caches. All of Sarah's traffic was delivered to the attacker first; with IP forwarding enabled they relayed it onward so everything appeared normal to Sarah.

Because http://mycampus.com carries no encryption, every byte was readable in plaintext. The attacker captured Sarah's login credentials and — critically — the session cookie issued by the server after login.

Q2 — The ChainThe session cookie was enough — no password required
How sessions work: After login the server creates a session and issues the browser a cookie — a token meaning "whoever holds this is authenticated as Sarah." The browser attaches it to every subsequent request automatically.

On a plain HTTP connection that cookie travels in cleartext. The attacker read it, injected it into their own browser via developer tools, and reloaded the page. The server saw a valid session token and treated the attacker as Sarah. The actual password was never needed again. The attacker then changed the recovery email and triggered a password reset to their own inbox — locking Sarah out entirely.

Q3 — Developer FixesSix controls that would have stopped or contained the attack
  • Deploy HTTPS (TLS). Encrypts the entire connection. An intercepted session cookie is unreadable ciphertext. The highest-impact fix — free certificates available via Let's Encrypt.
  • Enforce HSTS. The Strict-Transport-Security header tells browsers to always upgrade to HTTPS for this domain — even if the user types http://.
  • Set Secure, HttpOnly, and SameSite cookie flags. Secure blocks cookie transmission over HTTP. HttpOnly blocks JavaScript from reading it. SameSite=Strict blocks CSRF.
  • Require current password before sensitive account changes. Any change to recovery email or password must demand re-entry of the existing password. A stolen session alone cannot complete the takeover.
  • Invalidate all sessions on account changes. When a password or email is updated, every active session token must be revoked server-side immediately.
  • Alert the previous address on account changes. An instant email — "your recovery email was changed, was this you?" — gives the real user a window to act before being locked out.

Q4 — User PrecautionsFive things Sarah could have done
  • Use a VPN on public Wi-Fi. Creates an encrypted tunnel from your device regardless of what the website does. Even with full ARP poisoning, the attacker intercepts only encrypted noise.
  • Verify HTTPS before entering credentials. The padlock and https:// confirm the connection is encrypted. On plain http://, never enter credentials on a shared network.
  • Enable Two-Factor Authentication (2FA). A stolen session cookie alone cannot change sensitive settings if a second factor is required.
  • Log out fully when done. A proper logout invalidates the session server-side. A stolen cookie for an invalidated session is worthless. Closing the browser tab is not logging out.
  • Treat unexpected security emails as emergencies. The "your email was changed" notification was the last chance to recover the account. Acting immediately from a trusted network could have prevented the lockout.

Bonus — ResponsibilityOne perspective on who bears more responsibility
The developer bears primary responsibility. Deploying HTTPS would have made the ARP poisoning entirely useless — the attacker would have captured only ciphertext. Free TLS certificates and standard guidance have been available since 2016. Leaving a login portal on plain HTTP is a fundamental failure of secure design.

Sarah's failure to use a VPN is a contributing factor — but it is unreasonable to expect every user to understand the security implications of http:// versus https://. Secure by default is the developer's responsibility. User precautions are a second layer of defence — not a substitute for the first.

A system should be safe for users who know nothing about security — because most users are exactly that.
